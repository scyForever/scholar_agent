# ScholarAgent 记忆管理机制

本文档说明当前项目中短期记忆、长期记忆和最终模型上下文的管理方式。

## 1. 总体目标

记忆系统解决两个问题：

- 保留近期对话细节：当前任务的约束、偏好、文件、上下文和追问不能丢。
- 压缩历史上下文长度：历史对话不能无限进入模型，需要分层摘要和预算裁剪。

当前链路如下：

```text
用户输入
  -> AgentV2 启动 trace 并绑定 LLM 调用上下文
  -> DialogueManager 更新短期记忆
     -> 近期原文层
     -> 重点提炼层
     -> LLM 摘要层
  -> MemoryManager 召回当前用户长期记忆
  -> MemoryContextBuilder 按预算构建 memory_context
  -> RuntimeGraph / Agent 使用 memory_context 生成回答
  -> MemoryManager 写入长期对话摘要
```

## 2. 短期记忆

短期记忆只存在于当前进程的会话态中，由 `DialogueManager` 管理，字段位于 `DialogueState.short_memory`。

### 2.1 原文层

原文层保存最近 `recent_raw_messages` 条消息，默认 6 条。

特点：

- 保留 `role/content` 原始消息。
- 优先服务当前轮追问、刚给出的格式约束和临时任务细节。
- 不保存全部历史，只保存最近窗口。

示例：

```text
短期记忆-近期原文层：
user：我偏好结构化回答，重点关注记忆评测。
assistant：好的，我会按问题、方法、验证组织。
user：继续完善短期长期记忆管理。
```

### 2.2 重点提炼层

重点提炼层从最近 `highlight_source_messages` 条消息中抽取高价值句子，默认最多 10 条。

当前提炼依据是本地规则，命中以下信号词会进入候选：

- 偏好
- 关注
- 需要
- 不要
- 研究
- 论文
- 方法
- 数据
- 结论
- 格式
- 时间
- 范围
- 主题

特点：

- 不调用 LLM，稳定且无额外成本。
- 只保存对任务执行有价值的约束、偏好和事实。
- 按最近优先去重并裁剪。

示例：

```text
短期记忆-重点提炼层：
- user：我偏好结构化回答，重点关注记忆评测
- user：不要保留旧版本数据库兼容逻辑
- user：需要保留近期对话细节，同时压缩历史上下文长度
```

### 2.3 摘要层

摘要层通过 `DialogueManager` 注入的 `LLMManager` 生成，默认最多保留 900 字符。它的职责是把近期原文窗口之外的历史压缩成可继续使用的短期背景，而不是把完整历史重新塞回 prompt。

生成输入包括：

- 待摘要的历史消息，格式为 `序号. role：content`
- 当前重点提炼层结果

摘要提示词约束：

- 只基于给定对话生成中文摘要。
- 保留用户目标、约束、偏好、已确认结论和待跟进点。
- 不补充外部事实。
- 不输出标题、JSON、Markdown 代码块或解释。
- 输出目标为 120 到 300 字，最终仍受 `max_summary_chars=900` 限制。

特点：

- 用于保留较早对话的任务背景和决策。
- 不保留完整历史正文。
- 摘要生成会走现有 LLM provider 与 fallback 链路；未配置真实模型时会落到 mock provider，保证链路可运行，但摘要质量不代表真实模型效果。
- 原文层和重点提炼层仍是确定性规则，只有摘要层使用 LLM。
- `AgentV2.chat()` 会先启动 trace 并绑定 LLM 上下文，再写入用户消息，因此摘要层 LLM 调用也会进入白盒 trace。

示例：

```text
短期记忆-摘要层：
用户正在完善 ScholarAgent 的记忆机制，核心约束是保留短期原文层和重点提炼层的稳定规则，只把摘要层替换为 LLM 摘要。当前已确认长期记忆继续按 user_id 隔离，最终上下文仍由 MemoryContextBuilder 按预算组合，后续检查应重点关注摘要层是否进入 memory_context 和 trace。
```

## 3. 长期记忆

长期记忆由 `MemoryManager` 管理，默认落盘到 `data/memory/memory.db`。

### 3.1 数据库 schema

当前版本只面向新库，`memories` 表结构如下：

```sql
CREATE TABLE IF NOT EXISTS memories (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    owner_key TEXT NOT NULL,
    type TEXT NOT NULL,
    content TEXT NOT NULL,
    keywords TEXT NOT NULL,
    metadata TEXT NOT NULL,
    importance REAL NOT NULL,
    access_count INTEGER NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
)
```

索引：

```sql
CREATE INDEX IF NOT EXISTS idx_memories_owner_type ON memories (owner_key, type);
CREATE INDEX IF NOT EXISTS idx_memories_owner_updated ON memories (owner_key, updated_at);
```

说明：

- `user_id`：调用侧传入的用户或会话标识。
- `owner_key`：由 `user_id` 哈希得到，用于用户隔离查询。
- `type`：记忆类型，例如 `preference`、`paper_summary`、`research_note`、`conversation`。
- `content`：可直接供模型使用的压缩文本。
- `keywords`：写入时自动抽取的关键词数组。
- `metadata`：论文 ID、标题、主题、来源等结构化补充信息。
- `importance`：重要度，参与排序。
- `access_count`：召回次数。
- `created_at / updated_at`：用于时间衰减和最近列表。

### 3.2 写入策略

长期记忆不保存全部原始对话，而保存压缩后的稳定信息。

常见写入入口：

- `remember_preference()`：保存用户偏好。
- `remember_paper()`：保存已读论文摘要。
- `store(..., memory_type=CONVERSATION)`：保存本轮对话摘要。

对话记忆写入格式：

```text
用户问题：...
任务意图：...
回答摘要：...
```

回答摘要会被截断，避免把完整长答案灌入长期记忆。

### 3.3 召回策略

长期记忆召回始终按用户隔离：

```sql
WHERE owner_key = 当前用户 owner_key
```

不会查询其他用户记录，也没有 `owner_key OR user_id` 旧兼容逻辑。

召回排序综合：

- 字符级 TF-IDF 相似度
- BM25
- 重要度
- 时间衰减
- 查询关键词重合
- 用户画像关键词重合

其中用户画像关键词来自当前用户的 `preference` 和 `research_note` 记录。

## 4. 预算化上下文构建

`MemoryContextBuilder` 负责把短期记忆和长期召回结果转换为最终发给模型的 `memory_context`。

默认总预算：

```text
max_chars = 5000
```

默认预算分配：

```text
近期原文层：45%
重点提炼层：25%
摘要层：15%
长期记忆：15%
```

构建顺序：

1. 从近期原文层选取最近消息，保留细节。
2. 从重点提炼层选取最近高价值要点。
3. 加入摘要层，保留较早历史背景。
4. 加入长期记忆 top 结果。
5. 如果整体超预算，执行字符级截断。

输出示例：

```text
短期记忆-近期原文层：
...

短期记忆-重点提炼层：
...

短期记忆-摘要层：
...

长期记忆-用户专属召回：
...
```

同时会在 trace 中记录预算统计：

```json
{
  "context_chars": 3120,
  "max_chars": 5000,
  "short_raw_used": 4,
  "short_highlights_used": 6,
  "short_summary_used": true,
  "long_used": 2
}
```

## 5. Agent 主链路

`AgentV2.chat()` 中的记忆流程：

```text
1. 启动 trace，并将 LLMManager 绑定到当前 trace
2. 当前用户消息写入 DialogueManager.history
3. DialogueManager 更新 short_memory：
   - 原文层保存最近窗口
   - 重点提炼层按规则抽取高价值句子
   - 摘要层调用 LLMManager 生成 LLM 摘要
4. MemoryManager 按 user_id 召回长期记忆
5. MemoryContextBuilder 构建 memory_context
6. trace 记录 memory_recall：
   - short_layers
   - long_count
   - context_budget
7. RuntimeGraph 使用 memory_context
8. 回答完成后写入长期对话摘要
```

注意：

- 发给模型的不是全部历史，而是预算化后的 `memory_context`。
- 近期细节优先保留，远期历史压缩为重点和摘要。
- 长期记忆只召回当前用户自己的记录。

## 6. 评测覆盖

`data/evaluation/agent_eval_dataset.json` 当前包含 120 条 agent case，其中 20 条是记忆专项 case。

记忆专项 case 覆盖：

- 短期原文层是否存在。
- 重点提炼层是否存在。
- 摘要层是否存在。
- 当前用户长期记忆是否能召回。
- 其他用户长期记忆是否被隔离。

`ProjectEvaluationRunner` 会为评测构建临时记忆库：

```text
cache/evaluation_runtime/agent/memory_eval.sqlite
```

因此评测不会污染真实 `data/memory/memory.db`。

## 7. 维护建议

推荐约束：

- 不把完整长答案写入长期记忆。
- 不把全部历史消息发给模型。
- 新增记忆类型时，同步更新召回过滤和评测 case。
- 调整预算时，同步检查 `memory_recall.context_budget`。
- 调整摘要层 prompt 时，只改 LLM 摘要层，不改变原文层和重点层的稳定规则。
- 若摘要质量异常，先检查 `llm` trace 中 `purpose=short_memory_summary` 的 provider、prompt preview、response preview 和错误信息。
