# 面试速记版

## 1. 一句话介绍

ScholarAgent 是一个面向学术研究场景的多智能体助手，支持多源论文检索、本地 PDF 建库、综述写作、代码生成、长期记忆和白盒执行追踪。

## 2. 当前版本的核心亮点

- 任务分层：先根据意图、槽位和问题复杂度生成 `TaskConfig`
- 多智能体链路：`search / analyze / debate / write / coder`
- 混合检索：外部学术搜索 + 本地 RAG
- 推理引擎：7 种模式，其中 `ReAct / ToT / Debate` 已经是多步真实流程
- 可观测性：trace 落盘，前端执行时间线可直接看到 LLM 用途、provider、model、阶段模型
- 前端状态恢复：改主题、后退、刷新后可恢复最近聊天记录

## 3. 端到端主链路

用户问题进入系统后，主流程是：

1. `memory_recall`：回忆最近相关历史
2. `intent`：识别任务类型
3. `slots`：抽取主题、时间范围、最大论文数等槽位
4. `planning`：生成任务等级、允许推理模式、调用预算
5. `runtime_graph`：根据意图和模式调度 `search / analyze / debate / write / quality`
6. `trace`：把每一步持久化到 `logs/traces`

## 4. 你可以重点讲的设计点

### 4.1 搜索链路

- 外部搜索不是直接把用户原问题丢给数据库
- 系统会先做 LLM 查询改写，生成 `external_queries` 和 `local_queries`
- 搜索工具规划通过 `LangChain agent` 执行，但当前运行时固定只用 `zhipu`
- 如果 `zhipu` 不可用，自动退回确定性多源检索

### 4.2 本地 RAG

- 上传 PDF 后会抽取全文、表格、QA 和图谱片段
- 数据会写入 `SQLite + ChromaDB`
- 检索时走 `TF-IDF/BM25 + 向量检索 + RRF + BGE-Reranker + 相关性判断`
- 外部搜索到的论文不会自动写进本地 RAG

### 4.3 推理引擎

支持 7 种模式：

- `direct`
- `cot`
- `react`
- `tot`
- `debate`
- `reflection`
- `cove`

其中当前最值得讲的是：

- `react`：模型逐轮输出 JSON 决策，再真实调用工具，再根据观察继续推理
- `tot`：显式扩展多个候选分支，逐层打分并做 beam 剪枝
- `debate`：正方立论、反方质询、正方复辩、主持裁决四轮对辩

### 4.4 前端可观测性

- 前端右侧有执行时间线
- `llm` 步骤可看到 `purpose / provider / model / latency / error`
- `analyze / reasoning / debate / write` 阶段卡片会额外显示这一阶段实际使用的模型
- 页面刷新、切主题或后退时，前端会尝试恢复最近历史

## 5. 高频面试问答

### Q1：为什么要做多智能体，而不是一个大 prompt 全做完？

可回答为：

- 学术任务天然分阶段，检索、分析、综合、写作的输入输出不同
- 拆成多代理后，每一步更容易约束输入、预算和失败恢复
- trace 更清晰，出问题时可以直接定位是 `search`、`analyze` 还是 `write`

### Q2：搜索工具规划是做什么的？

可回答为：

- 它负责在外部学术检索前选择应该调用哪些搜索工具，以及怎么组合查询
- 这一步不是最终写作，而是检索策略规划
- 当前运行时固定只让 `zhipu` 执行这一步，避免多个 provider 在 agent/tool-calling 兼容性上表现不一致

### Q3：为什么不是所有搜索结果都进入本地 RAG？

可回答为：

- 当前本地 RAG 只负责用户显式上传或显式建库的本地知识
- 外部搜索结果是会话级临时材料，不自动持久化到向量库
- 这样可以避免一次搜索就把噪声结果永久写入本地知识库

### Q4：推理模式是怎么选的？

可回答为：

- 先由规划器给出允许模式集合，比如 `["cot", "debate", "reflection"]`
- 再由推理引擎根据 query 关键词和任务特征做 `auto` 选择
- 例如“比较/对比”优先 `debate`，“综述”优先 `reflection`，“实现/流程”优先 `react`

### Q5：当前版本最有工程价值的能力是什么？

可回答为：

- 不是单一模型能力，而是“检索、推理、写作、追踪”这一整条链路已经打通
- 尤其是可观测性强，能直接从 trace 定位问题
- 对学术场景来说，本地 PDF + 外部搜索 + 多步推理的组合比单点功能更实用

## 6. 讲项目时建议主动提到的边界

- 搜索规划固定使用 `zhipu`，可用性会依赖它
- `ReAct / ToT / Debate` 虽然已是多步流程，但仍是单模型协调，不是多进程并行 agent
- 外部搜索结果不会自动写入本地 RAG
- 如果 provider 或外部学术源不稳定，系统会降级，但时延和效果会波动
