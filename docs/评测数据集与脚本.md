# ScholarAgent 评测数据集与脚本说明

## 1. 文档目的

这套评测集围绕 ScholarAgent 的真实目标来设计：

- 它是学术研究助手
- 它要围绕论文知识库完成检索增强
- 它要把检索过程和生成过程分别评估
- 它还要评估 agent 在论文任务流程中的编排能力

因此，评测对象不再是“项目内部某个类是做什么的”，而是“围绕论文知识库完成学术任务的能力”。

---

## 2. 当前数据规模

当前默认通过生成器自动构建：

- `120` 篇模拟论文语料
- `120` 条检索评测 case
- `120` 条生成评测 case
- `120` 条 agent 评测 case

也就是说，语料和测试用例都已经超过“至少 100 项”的要求。

  命令行入口。

## 4. 自建论文知识库设计

语料不是项目实现细节，而是论文知识库场景。当前覆盖这些主题：

- 混合学术检索与 RRF 融合
- 学术检索查询改写与术语规范化
- Cross-Encoder 学术搜索重排
- 学术 RAG faithfulness 评测
- 引用可追溯的综述写作
- 多智能体科学综述综合
- 长任务研究记忆
- 按时间线组织的综述规划
- 方法分类与 taxonomy 构建
- 文献证据验证与答案真实性
- 领域自适应学术检索
- 研究助手 benchmark 对齐

每个主题又扩展到多个学科场景，例如：

- 生物医学
- 机器人
- 气候科学
- 材料科学
- 教育技术
- 金融科技
- 法律科技
- 医疗健康
- 能源系统
- 农业科学

这样做的目标是：

- 让语料更像真实研究助手会遇到的论文集合
- 让检索 case 和生成 case 都围绕论文知识库展开
- 保证同一主题在不同学科上也能形成足够多样的测试样本

## 5. 为什么要把检索和生成分开评测

如果把检索和生成混在一起打一个总分，会有两个问题：

- 你无法判断失败是因为没找到对的论文，还是因为找到了却回答错了
- 你无法单独优化某一侧，例如先调召回，再调答案真实性

因此现在拆成三套报告：

- `retrieval_report.json`
  只看检索是否找对上下文。
- `generation_report.json`
  使用金标准上下文，只看生成是否忠于证据。
- `agent_report.json`
  看 agent 是否按预期理解任务、提取槽位、走对流程并稳定完成任务。

另外保留一个组合报告：

- `rag_report.json`
  仅用于汇总 `retrieval + generation` 两份结果。

## 6. 检索评测设计

### 6.1 评测目标

检索评测只回答一个问题：

系统是否把真正相关的论文找回来了。

### 6.2 输入数据

每个 retrieval case 包含：

- `case_id`
- `query`
- `top_k`
- `relevant_doc_ids`

当前 `120` 条 retrieval case 采用三档难度混合生成：

- `48` 条 exact title case
  - 查询中包含完整标题和关键实验提升，用于检验精确命中能力
- `36` 条 semantic pair case
  - 查询只给题名前缀、主题关键词和两个领域约束，用于检验前两位纯度
- `36` 条 semantic triple case
  - 查询给题名前缀、主题关键词和三个领域覆盖要求，用于检验多文档覆盖与排序纯度

### 6.3 指标与计算方式

#### `recall_at_k`

含义：

- 相关文档被找回的覆盖率。

公式：

```text
recall_at_k = |retrieved_doc_ids ∩ relevant_doc_ids| / max(|relevant_doc_ids|, 1)
```

#### `precision_at_k`

含义：

- 返回结果中真正相关文档的占比。

公式：

```text
precision_at_k = |retrieved_doc_ids ∩ relevant_doc_ids| / max(|retrieved_doc_ids|, 1)
```

#### `context_relevance`

含义：

- 实际返回上下文对查询的整体相关性。
- 默认优先使用 provider 大模型 API 作为裁判，按查询和已返回上下文判断主题匹配程度与噪声比例。
- 如果 provider 不可用，则回退到基于 gold 文档集合的规则分。

公式：

```text
context_relevance =
provider_judge(query, retrieved_contexts).context_relevance.score

fallback_rule_context_relevance =
count(doc_id ∈ relevant_doc_ids for doc_id in retrieved_doc_ids)
/ max(len(retrieved_doc_ids), 1)
```

## 7. 生成评测设计

### 7.1 评测目标

生成评测不再依赖实时检索结果，而是直接使用金标准上下文。

这样评的是：

- 在“上下文已经给对”的前提下，系统能不能写出真实、相关、忠于证据的回答。

### 7.2 输入数据

每个 generation case 包含：

- `case_id`
- `query`
- `context_doc_ids`
- `reference_points`
- `forbidden_keywords`

其中：

- `context_doc_ids` 是金标准上下文来源
- `reference_points` 是标准答案应该覆盖的事实点
- `forbidden_keywords` 用于检查明显幻觉或偏题内容

### 7.3 中间概念

在打分时会先抽出三个集合：

- `mentioned_facts`
  回答里被提到的参考事实点
- `supported_facts`
  金标准上下文里确实能找到支撑的事实点
- `hallucination_hits`
  回答里命中的禁止关键词

### 7.4 指标与计算方式

#### `faithfulness`

含义：

- 回答里提到的内容，有多少真的被上下文支持。
- 默认优先使用 provider 大模型 API 结合问题、金标准上下文、参考事实点和候选回答做裁判打分。
- 如果 provider 不可用，则回退到规则分。

公式：

```text
faithfulness =
provider_judge(query, context_documents, answer, reference_points).faithfulness.score

fallback_rule_faithfulness =
|mentioned_facts ∩ supported_facts|
/ max(|mentioned_facts| + |hallucination_hits|, 1)
```

#### `answer_truthfulness`

含义：

- 回答里没有被判定为显式幻觉的事实比例。

公式：

```text
answer_truthfulness =
provider_judge(query, context_documents, answer, reference_points).answer_truthfulness.score

fallback_rule_answer_truthfulness =
|mentioned_facts|
/ max(|mentioned_facts| + |hallucination_hits|, 1)
```

#### `answer_relevance`

含义：

- 回答覆盖了多少标准答案应包含的事实点。

公式：

```text
answer_relevance =
provider_judge(query, context_documents, answer, reference_points).answer_relevance.score

fallback_rule_answer_relevance =
|mentioned_facts|
/ max(|reference_points|, 1)
```

## 8. Agent 评测设计

### 8.1 评测目标

Agent 套件不只评“答得像不像”，更评：

- 是否识别了正确意图
- 是否提取了正确槽位
- 是否按预期走过 `search / analyze / write` 等流程
- 是否稳定地完成任务

### 8.2 当前任务类型

当前 `120` 条 agent case 覆盖五类任务：

- 只基于本地知识库解释论文或方法
- 比较两篇论文/两类方法
- 带结构化约束的综述生成
- 缺失 topic 时的追问补槽
- 记忆专项评测：短期记忆原文层、重点提炼层、LLM 摘要层是否被构建并进入 `memory_recall`
- 记忆专项评测：当前用户长期记忆是否能按关键词召回，其他用户长期记忆是否被 `owner_key` 隔离

### 8.3 核心指标与计算方式

#### `task_success_rate`

含义：

- 多次运行中任务真正成功完成的比例。

公式：

```text
task_success_rate = mean(1 if run.task_success else 0 for run in runs)
```

#### `function_match_score`

含义：

- 意图、槽位、trace、artifact、记忆行为等功能维度整体匹配程度。

公式：

```text
function_match_score =
mean([
  needs_input_match,
  intent_match,
  slot_coverage,
  slot_value_match,
  trace_step_coverage,
  forbidden_step_ok,
  missing_slot_match,
  artifact_match,
  memory_match,
  optional(semantic_answer_score)
])
```

#### `performance_stability`

含义：

- 重复运行时延和成功率的稳定程度。

当前输出：

- `latency_mean_ms`
- `latency_std_ms`
- `success_std`
- `function_match_std`

#### `process_metrics`

含义：

- 逐过程指标的平均值。

包括：

- `intent_match`
- `slot_coverage`
- `slot_value_match`
- `trace_step_coverage`
- `forbidden_step_ok`
- `missing_slot_match`
- `artifact_match`
- `memory_match`
- `answer_keyword_coverage`
- `trace_step_count`
- `local_hit_count`
- `constraint_budget_present`
- `evidence_priority_present`

## 9. 评测脚本运行方式

先激活环境：

```bash
source /home/a1/miniconda3/etc/profile.d/conda.sh
conda activate agent
```

### 9.1 重建数据集

```bash
python run_evaluation.py --rebuild-datasets
```

### 9.2 只跑检索评测

```bash
python run_evaluation.py --suite retrieval --metric-judge-mode provider
```

说明：

- 不建议再用 `--top-k 4` 覆盖默认值，因为 retrieval case 已经按 `1 / 2 / 3` 三档 `top_k` 设计，覆盖后会破坏 `precision_at_k` 的区分度。

### 9.3 只跑生成评测

```bash
python run_evaluation.py --suite generation --answer-source oracle --metric-judge-mode provider
```

### 9.4 只跑 Agent 评测

```bash
python run_evaluation.py --suite agent --repeats 1 --agent-llm-mode mock
```

### 9.5 跑组合报告

```bash
python run_evaluation.py --suite rag --answer-source oracle --metric-judge-mode provider
```

### 9.6 跑全部评测

```bash
python run_evaluation.py --suite all --answer-source auto --repeats 2 --agent-llm-mode auto --metric-judge-mode provider --enable-vector --enable-reranker
```

补充说明：

- `--metric-judge-mode provider`
  - 优先调用已配置 provider 的大模型 API 做指标裁判打分
- `--metric-judge-mode auto`
  - 有健康 provider 时走大模型裁判，否则回退规则分
- `--metric-judge-mode rule`
  - 只使用本地规则分，适合离线回归或成本敏感场景

## 10. 结果输出

评测完成后会生成：

- `reports/evaluation/retrieval_report.json`
- `reports/evaluation/generation_report.json`
- `reports/evaluation/agent_report.json`
- `reports/evaluation/rag_report.json`

其中每份报告都包含：

- `aggregate`
- `cases`
- `metric_formulas`
- `judge_sources`

这样在看分数时，不需要再去翻代码猜公式。

## 11. 当前实现取舍

- `retrieval` 默认关闭向量检索和 reranker，优先保证稳定可跑
- `generation` 默认建议用 `oracle`，用来隔离检索噪声
- `agent` 默认建议用 `mock` LLM，优先评流程、槽位和稳定性，而不是远程模型文风
- `metric_judge_mode=provider` 会引入真实 provider API 延迟与成本；如需稳定批量回归，建议切换为 `rule`

如果后续要继续扩展，建议优先做：

- 更大规模的主题和学科扩展
- 多跳证据组合任务
- 更细粒度的生成质量评测
- 基于真实 provider 的生成基线对比
