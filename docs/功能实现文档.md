## 2. 整体实现总览

ScholarAgent 当前的实现主线可以概括为：

```text
用户问题
  -> AgentV2
    -> 预处理：记忆召回 / 意图识别 / 槽位提取 / 任务配置
    -> RuntimeGraph
         -> MultiAgentCoordinator
              -> Planner / Search / Analyze / Debate / Write / Coder
         -> ReasoningEngine
         -> QualityEnhancer
    -> Trace 持久化
    -> 长期记忆存储
```

项目按四层组织：

- `tools`
  负责接入搜索、文档获取、PDF 解析、网页补充检索等原子能力
- `skills`
  负责把多个工具拼成稳定的研究动作
- `agents`
  负责在特定任务阶段调用 skill 或 tool，并组织输入输出
- `runtime`
  负责会话状态、流程编排、失败降级、质量增强和 trace

核心入口文件：

- [agent_v2.py](/media/a1/16T/lcy/scholar_agent/src/core/agent_v2.py)
- [runtime_graph.py](/media/a1/16T/lcy/scholar_agent/src/pipeline/runtime_graph.py)
- [graph.py](/media/a1/16T/lcy/scholar_agent/src/pipeline/graph.py)
- [multi_agent.py](/media/a1/16T/lcy/scholar_agent/src/agents/multi_agent.py)

## 3. 启动层与基础设施

### 3.1 启动入口

目的：

- 为项目提供统一启动点
- 支持 Web、CLI 和基础验证三种运行方式

实现位置：

- [run.py](/media/a1/16T/lcy/scholar_agent/run.py)

具体实现：

- `main()` 读取用户输入的模式编号
- `run_web()` 调用 `src/ui/gradio_app.py`
- `run_cli()` 直接实例化 `AgentV2` 并进入交互循环
- `run_verify()` 执行 `verify_features.py`

这层很薄，不承载业务逻辑，只做入口分发。

### 3.2 全局配置

目的：

- 集中管理路径、RAG 参数、provider 配置、模型路径与运行目录
- 保证各模块使用同一组基础配置

实现位置：

- [settings.py](/media/a1/16T/lcy/scholar_agent/config/settings.py)

具体实现：

- `AppSettings` 用 dataclass 保存所有全局配置
- `ensure_runtime_dirs()` 在启动时自动创建 `data/`、`logs/`、`cache/` 等目录
- 通过环境变量覆盖默认值，例如：
  - `RAG_VECTOR_COLLECTION`
  - `RAG_PARALLEL_WORKERS`
  - `BGE_M3_MODEL_PATH`
  - `BGE_RERANKER_MODEL_PATH`

这层解决的是“所有模块都要用到基础路径和运行参数，但不应该各自硬编码”的问题。

### 3.3 核心数据结构

目的：

- 统一项目内部各阶段传递的数据格式
- 避免出现“每个模块都自定义一套 dict”的混乱状态

实现位置：

- [models.py](/media/a1/16T/lcy/scholar_agent/src/core/models.py)

关键结构：

- `Paper`
- `PaperAsset`
- `ParsedDocument`
- `SearchResult`
- `PaperAnalysis`
- `ResearchPlan`
- `ReasoningResult`
- `AgentResponse`
- `DialogueState`
- `IndexedChunk`

这些 dataclass 构成了项目的“中间语言”。

### 3.4 LLM 管理

目的：

- 屏蔽不同 provider 的差异
- 统一普通调用、结构化输出、LangChain chat model、预算控制和 trace 记录

实现位置：

- [llm.py](/media/a1/16T/lcy/scholar_agent/src/core/llm.py)

具体实现：

- `LLMProvider` 定义统一抽象接口
- `LangChainOpenAICompatibleProvider` 适配 OpenAI-compatible 接口
- `MockProvider` 在未配置真实模型时提供兜底响应
- `LLMManager` 负责：
  - provider 注册与优先级
  - 成功/失败状态记录
  - 预算控制
  - 当前 trace / stage 绑定
  - 普通文本输出与结构化输出
  - LangChain tool-calling 所需的 chat model 构造

这层的设计重点是“远程模型可替换、可降级、可追踪”。

### 3.5 Prompt 模板管理

目的：

- 避免把提示词散落在多个 agent 中
- 支持默认模板落盘和按名称渲染

实现位置：

- [manager.py](/media/a1/16T/lcy/scholar_agent/src/prompt_templates/manager.py)

具体实现：

- `DEFAULT_TEMPLATES` 内置了常用模板：
  - `intent_classification`
  - `paper_analysis`
  - `survey_writer`
  - `paper_answer_writer`
  - `compare_writer`
  - `concept_writer`
  - `daily_update_writer`
- `ensure_default_templates()` 会把默认模板写到 `data/prompts/`
- `render()` 按模板变量自动渲染

## 4. 预处理层

预处理层的目标是：在真正进入搜索和写作之前，把自然语言请求转成系统可执行的结构化输入。

### 4.1 对话状态管理

目的：

- 保存当前会话的历史消息、缺失槽位和上轮搜索结果
- 为多轮追问提供上下文

实现位置：

- [dialogue_manager.py](/media/a1/16T/lcy/scholar_agent/src/preprocessing/dialogue_manager.py)

具体实现：

- 使用 `defaultdict(DialogueState)` 按 `session_id` 保存状态
- 维护：
  - `history`
  - `intent`
  - `current_slots`
  - `missing_slots`
  - `last_trace_id`
  - `last_search_result`

这个模块解决的是“会话内上下文延续”的问题。

### 4.2 意图识别

目的：

- 判断用户当前到底在做哪类任务
- 为后续流程选择正确的 agent flow

实现位置：

- [intent_classifier.py](/media/a1/16T/lcy/scholar_agent/src/preprocessing/intent_classifier.py)

具体实现：

- 第一层：规则匹配
  - 如“综述”“比较”“解释”“这篇论文”等
- 第二层：TF-IDF 字符级相似度
  - 基于 `INTENT_SPECS` 描述文本做相似度分类
- 第三层：LLM 结构化分类
  - 当本地结果不够可靠且存在可用 provider 时调用

设计意图：

- 规则命中快
- TF-IDF 便宜且稳定
- LLM 只在必要时介入

### 4.3 槽位提取

目的：

- 从自然语言中提取可执行参数
- 把“用户说的话”变成“系统能消费的字段”

实现位置：

- [slot_filler.py](/media/a1/16T/lcy/scholar_agent/src/preprocessing/slot_filler.py)

具体实现：

- 用正则和规则抽取：
  - `time_range`
  - `max_papers`
  - `comparison_target`
  - `context_source`
  - `rag_mode`
  - `topic` / `paper_title`
- `fill_slots_once()` 会检查当前意图需要的必填槽位是否齐全
- 若缺失关键信息，则直接生成追问而不进入下游流程

这个模块的作用不是“理解语义全部细节”，而是抽取后续流程最需要的约束条件。

### 4.4 查询改写

目的：

- 把自然语言问题改写成适合外部搜索和本地 RAG 的查询集合
- 支持中文主题向英文学术查询的转换

实现位置：

- [query_rewriter.py](/media/a1/16T/lcy/scholar_agent/src/preprocessing/query_rewriter.py)

具体实现：

- 先提取 `core_topic`
- 再生成：
  - `english_query`
  - `external_queries`
  - `local_queries`
- 若有可用 provider，则调用结构化改写输出
- 否则走启发式改写：
  - 术语翻译
  - survey / comparison 扩展
  - 清洗控制短语

这个模块同时服务两条链路：

- 外部学术搜索
- 本地 RAG

### 4.5 任务分级与预算配置

目的：

- 判断当前任务是轻任务还是重任务
- 为 runtime 和多智能体流程提供预算约束

实现位置：

- [task_hierarchy.py](/media/a1/16T/lcy/scholar_agent/src/planning/task_hierarchy.py)

具体实现要点：

- 根据 query、intent 和 slots 输出任务等级与 `TaskConfig`
- `TaskConfig` 会影响：
  - 是否开启多智能体
  - 最大 LLM 调用数
  - 优先推理模式

## 5. 运行时编排层

### 5.1 `AgentV2` 总控入口

目的：

- 串起记忆、预处理、runtime、trace 和长期存储
- 对外提供一个统一 `chat()` 接口

实现位置：

- [agent_v2.py](/media/a1/16T/lcy/scholar_agent/src/core/agent_v2.py)

具体实现：

- 初始化所有核心组件：
  - `LLMManager`
  - `WhiteboxTracer`
  - `MemoryManager`
  - `DialogueManager`
  - `IntentClassifier`
  - `SlotFiller`
  - `TaskHierarchyPlanner`
  - `HybridRetriever`
  - `ReasoningEngine`
  - `QualityEnhancer`
  - `MultiAgentCoordinator`
  - `AgentRuntimeGraph`
- `chat()` 的主要职责：
  - 获取会话状态
  - 记忆召回
  - 意图与槽位解析
  - 生成任务配置
  - 调用运行图
  - 保存 trace 与长期记忆

它是整个项目最核心的编排入口。

### 5.2 RuntimeGraph

目的：

- 统一控制多智能体、推理兜底和质量增强
- 在 `LangGraph` 可用时使用图编排，否则回退顺序执行

实现位置：

- [runtime_graph.py](/media/a1/16T/lcy/scholar_agent/src/pipeline/runtime_graph.py)

具体实现：

- 节点：
  - `multi_agent`
  - `reasoning`
  - `quality`
- 路由规则：
  - 多智能体已有答案则结束
  - 没有答案则转 `reasoning`
  - `FULL + enable_quality_enhance=True` 时再走 `quality`

这是项目的“总调度图”。

### 5.3 MultiAgentPipeline

目的：

- 把研究执行拆成阶段化流水线
- 让不同意图复用同一套节点，但以不同 flow 运行

实现位置：

- [graph.py](/media/a1/16T/lcy/scholar_agent/src/pipeline/graph.py)
- [multi_agent.py](/media/a1/16T/lcy/scholar_agent/src/agents/multi_agent.py)

具体实现：

- 固定节点：
  - `plan`
  - `search`
  - `analyze`
  - `debate`
  - `write`
  - `coder`
- 不同 intent 对应不同 flow
- `MultiAgentCoordinator._resolve_flow()` 还会根据：
  - `rag_mode`
  - `context_source`
  - `prior_search_result`
  动态修正流程

这个设计避免了“大 agent 一把梭”的不可控问题。

### 5.4 推理引擎

目的：

- 在多智能体没有直接产出答案，或需要更强综合推理时提供兜底

实现位置：

- [engine.py](/media/a1/16T/lcy/scholar_agent/src/reasoning/engine.py)

具体实现：

- 支持多种推理模式：
  - `direct`
  - `cot`
  - `react`
  - `tot`
  - `debate`
  - `reflection`
  - `cove`
- `reason()` 会根据 query 自动选择合适模式
- `react` 模式还能接白名单工具做真实工具调用推理

这层的重点是：把“回答生成”与“复杂推理策略”解耦。

### 5.5 质量增强

目的：

- 在高质量模式下对已有答案做后处理增强
- 降低单次回答的不稳定性

实现位置：

- [enhancer.py](/media/a1/16T/lcy/scholar_agent/src/quality/enhancer.py)

具体实现：

- `self_moa()`
  - 生成多个候选答案
  - 聚合成更强答案
- `mpsc_verify()`
  - 从理论、证据、边界三个角度做一致性验证
  - 用 TF-IDF 相似度估计一致性分数

这层只在 `FULL` 模式下作为后处理启用，不是默认主路径。

## 6. 工具层实现

工具层解决的是“系统能调用哪些原子能力”。

### 6.1 工具注册与 schema 管理

目的：

- 统一工具注册
- 自动生成 LangChain 可调用工具
- 避免 agent 直接接触原始 Python 函数

实现位置：

- [registry.py](/media/a1/16T/lcy/scholar_agent/src/tools/registry.py)

具体实现：

- `ToolDefinition` 描述工具名称、说明、参数和标签
- `ToolRegistry.register()` 同时保存：
  - Python 函数
  - 工具定义
  - LangChain `StructuredTool`
- `register_tool()` 装饰器供具体工具注册使用

### 6.2 白名单控制

目的：

- 控制不同 agent 能看到和能调用的工具集合
- 降低误调用和越权调用风险

实现位置：

- [manager.py](/media/a1/16T/lcy/scholar_agent/src/whitelist/manager.py)

具体实现：

- 默认白名单写入 `data/whitelist.json`
- 不同 agent 分配不同工具集，例如：
  - `search_agent` 只看搜索相关工具
  - `analyze_agent` 看 PDF 解析和阅读类工具
  - `write_agent` 默认不直接拿工具

### 6.3 学术搜索工具

目的：

- 统一封装不同学术平台的检索接口
- 把平台差异收敛成统一的 `Paper` 列表

实现位置：

- [research_search_tool.py](/media/a1/16T/lcy/scholar_agent/src/tools/research_search_tool.py)

具体实现：

- 定义 `AcademicSourceAdapter` 抽象
- 为不同平台提供 adapter：
  - `ArxivAdapter`
  - `OpenAlexAdapter`
  - `SemanticScholarAdapter`
  - `WebOfScienceAdapter`
  - `PubMedAdapter`
  - `IeeeXploreAdapter`
  - `GoogleScholarAdapter`
- 每个 adapter 负责：
  - 请求参数构造
  - 结果解析
  - 时间范围过滤
  - 基础相关性排序
- 最终统一映射为 `Paper`

### 6.4 文档获取工具

目的：

- 根据 DOI、arXiv ID、PMID、PMCID 获取 PDF 或 HTML
- 屏蔽不同来源的获取差异

实现位置：

- [research_document_tool.py](/media/a1/16T/lcy/scholar_agent/src/tools/research_document_tool.py)

具体实现：

- `PaperAcquisitionService.fetch()` 根据标识符类型路由到不同下载逻辑
- 支持：
  - arXiv
  - DOI
  - PMC
  - PubMed
- 对外返回统一的 `PaperAsset`

### 6.5 PDF 解析与定向阅读

目的：

- 把 PDF 转成后续可读、可检索、可精读的结构化内容

实现位置：

- [research_document_tool.py](/media/a1/16T/lcy/scholar_agent/src/tools/research_document_tool.py)
- [pdf_tool.py](/media/a1/16T/lcy/scholar_agent/src/tools/pdf_tool.py)

具体实现：

- `PDFReadingService.parse_pdf()`
  - 使用 `PyMuPDF` 读取页面结构
  - 提取行文本
  - 识别标题与章节
  - 判断单双栏版面
  - 对章节正文做递归层次分割：优先按段落切分，块过大时继续按换行、句子、分句、词和字符逐层细化，并保留重叠上下文
  - 构造 `ParsedDocument`
- `extract_visuals()`
  - 导出图片
  - OCR 识别
  - 近似生成 Markdown 表格或 LaTeX 公式
- `read_section()`
  - 按章节名做定向阅读
- `extract_pdf_text()`
  - 面向 RAG 建库，把解析结果转成 `chunks / tables / qa_pairs`
  - `chunks` 来源于章节级递归分块结果，避免简单定长截断过早打断语义边界

### 6.6 Web 补充检索

目的：

- 当本地 RAG 证据不足时补充网页摘要

实现位置：

- [web_search_tool.py](/media/a1/16T/lcy/scholar_agent/src/tools/web_search_tool.py)

在当前实现中，它主要作为 RAG 后置补强，而不是主检索入口。

## 7. Skills 层实现

Skills 层解决的是“高频研究动作复用”问题。

实现位置：

- [research_skills.py](/media/a1/16T/lcy/scholar_agent/src/skills/research_skills.py)

### 7.1 `ResearchPlanningSkill`

目的：

- 把一个大研究主题拆成可执行的子任务

具体实现：

- `plan()` 根据 topic、intent、slots 生成 `ResearchPlan`
- 固定生成 6 个阶段任务：
  - 范围界定
  - 多源检索
  - 初筛聚类
  - 精读核心论文
  - 提炼共识与分歧
  - 输出最终交付
- 每个任务都带：
  - `goal`
  - `deliverable`
  - `dependencies`
  - `suggested_tools`

### 7.2 `LiteratureSearchSkill`

目的：

- 统一多源搜索，并结合研究记忆进行排序

具体实现：

- 内部调用 `search_platforms()`
- 如果带 `user_id`，会：
  - 优先把未读论文排前
  - 记录检索偏好

### 7.3 `DeepReadingSkill`

目的：

- 把论文获取、PDF 解析、章节阅读和图表提取做成复用动作

具体实现：

- `fetch_full_text()`
- `parse_pdf()`
- `targeted_read()`
- `extract_visuals()`

### 7.4 `ResearchMemorySkill`

目的：

- 保存用户偏好、已读论文和研究上下文
- 在搜索排序和上下文召回时复用

具体实现：

- `remember_preference()`
- `remember_paper()`
- `recall_context()`
- `rank_unseen_first()`
- `remember_search_preferences()`

Skills 层的价值在于：让这些动作可以被多个 agent 共享，而不是分散写在每个 agent 内部。

## 8. Agents 层实现

Agents 层解决的是“谁在什么阶段做什么事”。

### 8.1 研究专用 agent

实现位置：

- [research_agents.py](/media/a1/16T/lcy/scholar_agent/src/agents/research_agents.py)

包含：

- `ResearchPlannerAgent`
- `ResearchSearchAgent`
- `ResearchReadingAgent`
- `ResearchMemoryAgent`

它们的主要作用是：对 skill 做一层带 trace 的角色化封装。

### 8.2 `SearchAgent`

目的：

- 决定本轮检索到底怎么做
- 串起会话上下文、本地 RAG、外部搜索、排序和 trace

实现位置：

- [search_agent.py](/media/a1/16T/lcy/scholar_agent/src/agents/search_agent.py)

具体实现：

- 先读取 `topic / time_range / max_results / context_source / rag_mode`
- 处理三种特殊控制：
  - 复用上轮搜索结果
  - 只查本地 RAG
  - 完全跳过检索
- 默认先调 `HybridRetriever.retrieve()`
- 再决定是否补外部学术搜索
- 外部搜索有两种路径：
  - `LangChain agent` 规划工具调用
  - 确定性 fallback 搜索
- 最后：
  - 聚合去重
  - 记忆排序
  - 生成 `SearchResult`
  - 记录详细 trace

这是项目里最复杂、最关键的 agent。

### 8.5 `WriteAgent`

## 9. 本地 RAG 实现

RAG 是项目中最完整的一条功能链路之一。

实现位置：

- [retriever.py](/media/a1/16T/lcy/scholar_agent/src/rag/retriever.py)
- [vector_store.py](/media/a1/16T/lcy/scholar_agent/src/rag/vector_store.py)
- [bge_m3_embedder.py](/media/a1/16T/lcy/scholar_agent/src/rag/bge_m3_embedder.py)
- [bge_reranker.py](/media/a1/16T/lcy/scholar_agent/src/rag/bge_reranker.py)

### 9.1 建库目的

- 让本地已有 PDF 可以被后续问答复用
- 让系统在解释概念、分析论文、追问局限时优先使用本地证据

### 9.2 建库实现

- `index_pdf()`
  - 解析 PDF
  - 构造 `text_chunk / table_chunk / qa_chunk / kg_chunk`
  - 写入 SQLite
  - 写入 Chroma
  - 若向量入库失败则回滚 SQLite

### 9.3 检索实现

- 对话增强
- 查询改写
- 根据 query 路由 chunk 类型
- 并行执行：
  - SQLite 词法召回
  - Chroma 稠密召回
- `RRF` 融合
- `BGE Reranker` 重排
- LLM 相关性校验
- 命中不足时补网页摘要

### 9.4 RAG 与主链路的关系

- `SearchAgent` 默认先查本地 RAG
- `WriteAgent` 会直接消费 RAG 片段
- `SlotFiller` 允许用户用自然语言控制 RAG 行为
- `DialogueManager` 支持复用上轮检索结果

也就是说，RAG 在项目里不是独立插件，而是搜索与回答链路中的核心部分。

## 12. 一次完整请求是如何落地的

以用户输入：

```text
根据之前查找到的资料，解释多模态 RAG 的主要挑战
```

为例，落地过程如下：

1. `AgentV2.chat()`
   读取会话状态和上轮 `SearchResult`
2. `MemoryManager.recall()`
   召回与当前 query 相关的长期记忆
3. `IntentClassifier.classify()`
   判断当前意图是 `explain_concept`
4. `SlotFiller.fill_slots_once()`
   提取出：
   - `context_source=previous_search`
   - `topic=多模态 RAG 的主要挑战`
5. `TaskHierarchyPlanner`
   输出本轮任务配置
6. `RuntimeGraph.execute()`
   进入多智能体主链路
7. `MultiAgentCoordinator._resolve_flow()`
   根据 `context_source=previous_search` 保留 `search -> write`
8. `SearchAgent.run()`
   直接复用上一轮 `SearchResult`
9. `WriteAgent.run()`
   从：
   - 本地 RAG 片段
   - 补充网页片段
   - 论文列表
   组装材料并写出最终答案
10. `WhiteboxTracer`
    把全过程写入 trace
11. `MemoryManager.store()`
    把本轮对话写入长期记忆

这个例子体现了项目当前实现的一个核心特点：

- 它不是“每轮都重新搜一遍”
- 而是“先理解用户控制意图，再决定复用、检索还是跳过检索”
